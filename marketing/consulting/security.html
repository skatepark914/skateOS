How we handle security

Concrete practices, plain English. What we actually do for every client — reviewed by Jon (CISSP, OSCP) on every Production Sprint.

The short version

  • Hardware MFA on every account that touches your systems.
  • Encryption everywhere — TLS in transit, AES-256 at rest, on every database we touch.
  • Least privilege — we ask for the smallest amount of access we need, and we revoke it when we're done.
  • No credit cards on our servers — payment processors handle card data in their iframes, we never touch it.
  • Row-Level Security at the database layer for multi-tenant systems, so a bug in our code can't leak one customer's data to another.
  • Secrets stay in secret managers — never in code, email, Slack, or screenshots.
  • Backups that we've actually restored — a backup you've never tested is a hope, not a backup.

The rest of this page is the longer version of each.

Access

When we work on your systems we get accounts in your name, not shared logins. Hardware MFA (YubiKey or equivalent) on every account. We ask for the minimum permissions we need for the actual work — read-only when we can, write only on the tables/services we're actively changing.

When we're done, all access is revoked within 24 hours and credentials are rotated. You get a quick note confirming it's off.

Data & encryption

Everything in motion is over TLS 1.2 or higher (we disable older versions where we can). Everything at rest is encrypted at the storage layer — AES-256 on the database, AES-256 on object storage, encrypted backups with a separate key.

Sensitive data (payment cards, health info, anything regulated) gets stricter handling on a per-project basis. Payment card data never touches our infrastructure — we integrate with payment processors (Helcim, Stripe) that capture cards in their own iframe, and our database stores only their transaction reference.

At engagement end your data goes back to you and we destroy our copies within 30 days, or sooner on request. Backups containing your data wash out of our retention window within 90 days.

Infrastructure

We deploy to providers we trust and can audit: Cloudflare, AWS, Supabase, Vercel. No shared hosting, no random VPS providers we can't verify.

  • Default-deny firewall — production databases aren't reachable from the public internet, only from app servers.
  • WAF via Cloudflare on every public-facing site (OWASP Core Rule Set).
  • Row-Level Security at the database layer for multi-tenant systems — Supabase's RLS or equivalent. Even if app-layer auth has a bug, the database refuses cross-tenant reads.
  • Webhook signature verification on every incoming webhook so attackers can't fake them.
  • Content Security Policy headers on production HTML to reduce XSS blast radius.

Secrets

API keys, database passwords, signing tokens — stored only in real secret managers (Supabase secrets, AWS Secrets Manager, Cloudflare Workers secrets, 1Password). Never in code, config files, env files committed to git, email, chat, or screenshots.

Rotated when they should be rotated (quarterly minimum, immediately on suspected exposure). Audit-style git-history reviews surface accidentally-committed secrets that older codebases often carry — when we find them in client systems, we rotate first and document the finding for the audit trail.

Logging & monitoring

For projects we operate, we log the things that matter: authentication events, privileged actions, database changes on critical tables (with actor identity + old/new values), webhook deliveries, application errors.

Logs are kept long enough to be useful (90 days minimum, longer on request). Critical alerts page us via email + SMS. We're not a 24/7 SOC — if you need that, you need a different vendor — but for a side-project-scale problem on a non-critical system, we'll have eyes on it within hours, not days.

Backups

For production databases: continuous WAL streaming plus daily snapshots. For object storage: versioned and cross-region replicated where the cost makes sense. For infrastructure: everything in version control.

We test restores periodically. A backup that's never been restored is a hope, not a backup.

If something goes wrong

If something looks wrong — confirmed breach, suspicious activity, anything that might be an incident — we'll:

  • Acknowledge it fast (hours, not days).
  • Contain it — revoke credentials, isolate systems, whatever stops the bleeding.
  • Fix the root cause.
  • Tell you what happened, when, and what we did about it, in writing.

For regulated data (HIPAA, GDPR) we follow the applicable notification timelines. Otherwise, we tell you as soon as we know there's something to tell.

Contract-layer security · AI arbitration

Security isn't just code and infrastructure — it's also what happens when a customer disputes a charge, a user claims an injury, or a vendor sues. We bake AI arbitration into client contracts via our partnership with DecisionLayer (built by Liberty McAteer). Disputes get resolved in ~45-90 days for ~$500 instead of $100k+ over 18+ months of court.

It's standard inclusion on every Production Sprint where the build has a customer-facing contract, waiver, or terms of service. We've already piloted the clause against our own waiver (simulated $35k injury claim — system ruled for the operator). The point: we ship clients out the door with a contract stack that handles disputes the new way, not the 2002 way. See the DecisionLayer integration in our work portfolio.

Scope of what we offer

Worth being clear about what we do — and what's outside our scope so nobody buys the wrong thing:

  • In scope: Application-layer security reviews on every Production Sprint we deliver. Standalone Security Look-Over packages on existing codebases. Configuration audits, secrets-management hygiene, RLS / authentication / authorization review. HIPAA-aligned controls and BAAs on a per-engagement basis when in scope.
  • Out of scope: 24/7 managed SOC operations, follow-the-sun on-call rotations, formal third-party penetration-test reports with auditor sign-off. If you need those, we'll point you to a managed-services firm or an accredited pen-test vendor.

If you're not sure whether your project fits, send us a short note and we'll tell you straight.

The problem

Tree-care and outdoor-service companies are some of the most under-served SMBs in software. The market leaders look like this:

  • Jobber — generic field-service SaaS at $200-400/mo. Reasonable software, but every tree-care company ends up bolting on 5+ workarounds because Jobber doesn't know what an ISA certification is, doesn't track chemical-applicator licenses, doesn't understand crew-by-property history, doesn't have map-based equipment placement.
  • ServiceTitan — enterprise-priced ($500-2000/mo) and built for HVAC / electrical contractors. Tree-care workflows force-fit awkwardly. Multi-month implementations.
  • QuickBooks + paper / spreadsheets — what most independent tree-care companies actually use. Quotes scribbled on paper, photos texted to the office, invoices generated by hand, AR aging on a sticky note.

The combination is roughly: $400-600/mo of SaaS subscriptions that don't quite fit, plus 4-6 hours/week of admin work that wouldn't exist with the right tool. For a 3-6 person crew running $500k-2M/yr in revenue, that's tens of thousands per year of friction.

Field crews live in their trucks. Operators live in their phones. Neither lives at a desk. Software built for HVAC dispatchers misses this basic shape — and the gap shows up as paper, spreadsheets, and Saturday-morning catch-up work.

What we built

Branch Manager is a multi-tenant SaaS operations platform built specifically for outdoor-service crews — tree care, landscaping, lawn care, pest control. One platform that replaces the Jobber + QuickBooks + Smartwaiver + DocuSign + scattered-paper stack with an integrated flow from quote → job → invoice → recurring service.

Core flow

  • Quote-to-invoice — Crew lead snaps photos, sketches the property, marks where equipment goes. Quote generated, emailed, customer reviews on a mobile-friendly page, approves, optional deposit charged via Stripe.
  • Crew dispatch + scheduling — Schedule view of upcoming jobs, drag-and-drop assignment to crews, real-time job status as crews move through the day, automated email/SMS reminders to customers 24 hours out.
  • Mobile crew app — Crew foreman has the day's jobs on their phone, offline-first capture so they can update status from rural service areas without cellular. Photos, time tracking, completion notes, customer signature.
  • Customer self-serve portal — Authenticated customer page showing service history, upcoming jobs, invoices, autopay, the property record.
  • Stripe Connect + autopay — Recurring service customers go on autopay. Card on file, monthly bill, no chasing AR.
  • Vehicle inspection + safety compliance — Pre-job inspection logs, equipment maintenance tracking, ISA cert renewals, applicator license renewals.
  • Communications hub — All customer messages in one place. Email, SMS, voicemail transcription. Per-customer history.
  • Payroll + time tracking — Crew time entries by job, approval workflow, gross-pay calc with overtime, CSV export to your accounting system.

Beyond the core

Branch Manager has 22+ admin modules covering employee onboarding (incident reports, employment agreements, safety training certificates, wage-theft notices), expense tracking, campaigns + marketing, automations, command-palette navigation, custom form builder for service-specific intake, PDF generation for invoices and certificates, and a notifications system that ties everything together.

Multi-tenant from day 1

Branch Manager was built multi-tenant from the start — RLS-enforced at the database layer, tenant-scoped routing, owner / staff / crew / customer permission tiers. Adding a second tenant is a row in the tenants table, not a forklift migration. Multiple tree-care companies could be onboarded to the same install without code changes.

This architecture is what made skateOS possible — when Doug decided to build a recreation-industry parallel in 2026, the entire multi-tenant infrastructure ported over with just the domain layer rewritten.

Security choices & why

RLS at the database layer

Every business table has Row-Level Security policies tied to tenant_id and the authenticated user's role. Even if application-layer auth fails, the database refuses cross-tenant reads/writes. The customer portal is similarly scoped — a signed-in customer can read only their own records, enforced at the DB.

Stripe Connect = no card data on our servers

All payment captures happen via Stripe's hosted forms. We store only the Stripe customer ID and the latest charge reference — never the card PAN, CVV, or expiration. PCI scope = SAQ-A.

Webhook signature verification

Stripe webhooks verify the shared-secret signature before acting on any payload. Idempotency keys prevent replay attacks. Failed signature = silent 200 OK back to Stripe so attackers can't probe.

Audit log on every business table

Postgres trigger captures every INSERT/UPDATE/DELETE on core tables with actor identity. Owner can review changes by table/actor/date for forensic / dispute purposes.

Authentication via Supabase Auth

Magic-link sign-in for customers (no password storage). Hardware MFA option for owners. SSO via Google Workspace for staff.

Encrypted in transit + at rest

TLS 1.2+ everywhere via Cloudflare. AES-256 at rest via Supabase defaults. Backups encrypted with separate key.

How we shipped this fast

Branch Manager was built and put into production in months, not years. The shape of the work:

  • Pattern from the parent codebase — Branch Manager started as a fork of an earlier internal admin tool (Smart Lawn NY). The multi-tenant foundation, auth flow, payment integration, and audit log were already battle-tested before tree-care-specific work began.
  • Build at our own business first — Every feature ships into 2nd Nature Tree's production operation before it goes into the platform. If it doesn't work for our own crews, it doesn't ship. Real-world stress test before any customer sees it.
  • AI as force multiplier — Claude-assisted development collapsed the "write a new admin page" cycle from days to hours. Without AI velocity, Branch Manager doesn't exist as a 2-person-built product.
  • Vanilla JS for the admin — No framework churn. The admin SPA is plain HTML/JS/CSS. Every dependency added is justified individually.

Daily operations · front desk

Selling a membership to a new walk-in

  1. POS → click + next to customer dropdown → quick-add modal
  2. Name + DOB + email + phone + optional address. Guardian fields auto-appear if DOB < 18.
  3. Save → customer created
  4. Search for "Monthly Unlimited" (or chosen plan) → click to add to cart
  5. Helcim charge button → customer taps card → sale completes
What happens automatically next:
  • Subscription row inserts with status='active'
  • Trigger flags the customer for Brivo sync
  • Within 5 min the cron creates a Brivo user → adds to active-members group → sends Mobile Pass invite
  • skateOS sends the branded "🔓 You're in" welcome email
  • Customer installs Brivo app → walks to door → taps → 24/7 access from that moment

Selling a day pass

Same flow but pick "Day Pass — Adult" or "Day Pass — Under 12". No Brivo provisioning — day passes don't grant 24/7 access. Customer checks in at the front desk on each visit.

Selling a punch card

"10-Pack Punch Card" → ring up like any product. Punches decrement on each check-in. When count hits ≤2, Check-In auto-prompts the cashier with a renewal modal. No Brivo access (punch cards are session-based, not time-based).

Resending a Brivo invite

Customer detail → Brivo panel → "Resend mobile pass" button. Brivo re-fires the invite email.

Resending the welcome email (the skateOS branded one)

Customer detail → Brivo panel → "Resend welcome email" button. skateOS fires the rose-themed how-it-works email.

Banning a customer

  1. Customer detail → red "Ban" button at the bottom
  2. Type reason → confirm
What happens automatically: Tags get banned → trigger flags for Brivo sync → within 5 min credential is revoked → next phone tap returns access_denied → customer panel flips to 🔴 REVOKED with reason "banned".

Daily operations · members

A new member's first 24 hours

  1. Sign up at skateos.com/2ntr → "Become a member" → fills form at /admin/join.html
  2. Cashier rings up membership at POS → payment → auto-provision triggers
  3. Within 5 minutes: receives Brivo Mobile Pass invite (from Brivo) + skateOS welcome email "🔓 You're in"
  4. Installs Brivo Mobile app → accepts the pass
  5. Walks to park door → taps phone → door clicks open
  6. skateOS auto-creates a checkins row → appears in "in the park" list

Member self-serve portal · me.html

  1. Customer goes to app.skateos.com/admin/me.html
  2. Enters email → gets a magic-link email
  3. Clicks link → sees their portal:
    • Active passes + punches remaining
    • Loyalty points + tier badge
    • Recent purchases + receipt CSV download (year-end taxes)
    • Upcoming lessons + Add-to-Calendar buttons
    • Achievement badges
    • "✏️ Edit my profile" — update email/phone/guardian
    • "Pause membership" / "Cancel" (self-serve)

Phone dead at the door

  1. Press the intercom button on the BDS reader
  2. Front desk gets a buzz during staffed hours → buzzes them in
  3. Outside staffed hours, phone is the only way in (intentional — encourages members to keep phones charged)

Birthday parties + event passes

Quoting + booking a party

  1. Admin → Tools → "Party Quote" (opens admin/tools/party-quote.html)
  2. Fill: parent name/phone/email + skater name + DOB + party date+time + package + add-ons
  3. Quote shows total + 50% deposit
  4. Submit → creates customer + lesson row (type=birthday) + deposit invoice
  5. Click "Send deposit payment link" → Helcim hosted payment URL → text/email parent
When the parent pays the deposit:
  • Helcim webhook fires → flips invoice to paid
  • helcim-webhook auto-fires brivo-issue-event-pass for the party window (scheduled_at − 30 min to scheduled_at + duration + 30 min)
  • Parent gets Brivo Mobile Pass for the party time window only
  • Door auto-revokes after the party ends (cron sweeps every 5 min)
  • Buffer covers early setup + late cleanup

Manual event pass (industry comp / press visit / etc.)

  1. Customer detail → Brivo panel → "🎟️ Issue event pass" button (owner-only)
  2. Set window (default: next Saturday 2pm–6pm) + reason
  3. Save → pass appears as 🟢 LIVE NOW or ⏰ SCHEDULED
  4. Customer gets Brivo Mobile Pass for that window

Revoking an event pass early

Customer detail → Brivo panel → find the pass row → click "Revoke". Pass flips to revoked; access removed within 5 min (or instant via "Sync now").

Emergency procedures

Active incident (serious injury / fight / fire / intruder)

  1. Settings → Integrations → "🚨 Emergency lockdown" card
  2. Click "🔒 ENGAGE LOCKDOWN"
  3. Type reason ("active medical emergency" / "altercation at front desk")
  4. Type LOCKDOWN in all caps to confirm
What happens:
  • Every member's Brivo credential removed from active-members group in <30 seconds
  • Next phone tap returns access_denied
  • Sticky red banner across every admin page showing reason + actor + revoke count
  • Every customer's Brivo state shows 🟠 SUSPENDED

Ending lockdown

  1. Click "End lockdown · restore member access" (banner OR Settings card)
  2. Every customer with a Brivo credential re-flagged for sync
  3. Within 5 min cron re-evaluates each (eligible members restored; lapsed/banned stay revoked)

Suspected tailgater

  1. You'll get a Team Chat reminder when 3+ person events at park door happen without matching Brivo grants
  2. Investigate: Activity Log → Brivo events → filter by "Access denied"
  3. Check Dashboard → Park vision widget → "Latest detections" should show person events with snapshot links
  4. If known customer with lapsed credential, they need to renew
  5. If unknown, consider raising door capacity threshold OR adding guest-pass workflow

Park-at-capacity warning

When a Brivo grant fires while at settings.maxCapacity, a Team Chat reminder posts. Front desk should ask new arrivals to wait, or comp them a day pass for a slower time. Idempotent per hour.

White-label · onboarding a second park

When the second skatepark signs up for skateOS:

Step 1 · Provision them in Supabase

  1. Create their tenant row + user_tenants row for their owner email
  2. Their admin login lands on app.skateos.com with a clean Dashboard

Step 2 · Their owner configures their Brivo

  1. Settings → Integrations → "Brivo per-tenant credentials" card
  2. Click "Save credentials" → modal opens
  3. Paste their own Brivo OAuth client_id, client_secret, api_key, account_id
  4. Paste Brivo dashboard config IDs (active members group, park door AP, shop door AP, hours schedule)
  5. Generate + paste a webhook secret (openssl rand -hex 32)
  6. Save → click "Test credentials" → verifies OAuth + a real API call → stamps last_verified_at

Step 3 · Per-tenant webhook URL

  1. Settings card shows a "🔗 PER-TENANT WEBHOOK URL" block with their tenant UUID baked in
  2. Copy that URL → paste into THEIR Brivo dashboard → API Management → Webhooks
  3. From now on Brivo posts route directly to their tenant's webhook secret for signature verification

Step 4 · Rest of integrations

  • Helcim (their account, their API token) — Settings → Helcim card
  • Smartwaiver (their API key) — Settings → Smartwaiver
  • Resend (their sending domain) — Settings → Email
  • Twilio (their phone number) — Settings → SMS
  • Optional: skateOS Vision Box (Phase 9 — needs hardware shipping)

Step 5 · Park hours + branding

  • Settings → Park Hours → set their open/close per day
  • Settings → Business → their park name, phone, address, logo URL, theme color

Step 6 · Verify

  1. Dashboard → Setup Status card → progress bar should be 100%
  2. Invite a test member → see Brivo invite fire → walk to door → unlock
2nd Nature's env-var setup keeps working unchanged. Each park is fully isolated: their lockdown affects only their group, their members provision against only their Brivo account, their door events route to only their webhook secret.

State machine · who gets access

The brivo_member_desired view computes should_have_access per customer. Order of evaluation (first match wins):

ReasonTriggers FALSE
bannedCustomer has tag banned / do_not_serve / donotserve / 86d
no_waivercustomers.waiver_signed_at is NULL
waiver_expiredwaiver_expires_at < now()
no_active_membershipNo active monthly/annual subscription within end_date + not paused
event_pass (TRUE path)Active event pass window covers now()
eligible (TRUE path)Active membership + everything else clean

Common SQL

Who should have access right now + why 
SELECT name, should_have_access, desired_reason
FROM brivo_member_desired
ORDER BY should_have_access DESC, name;
Flagged customers waiting on sync 
SELECT id, name, brivo_credential_state, brivo_sync_needed_at, brivo_sync_error
FROM customers
WHERE brivo_sync_needed_at IS NOT NULL
ORDER BY brivo_sync_needed_at;
Recent access events for one customer 
SELECT occurred_at, access_point, event_type
FROM brivo_access_log
WHERE customer_id = '<uuid>'
ORDER BY occurred_at DESC
LIMIT 20;
Force-flag one customer for immediate sync 
SELECT brivo_flag_customer_sync('<uuid>');
View pg_cron schedule 
SELECT jobid, jobname, schedule, command
FROM cron.job
WHERE jobname LIKE 'brivo-%';
Brivo grants without matching person detection (tailgater audit, requires Vision Box) 
SELECT b.occurred_at, c.name
FROM brivo_access_log b
LEFT JOIN customers c ON c.id = b.customer_id
LEFT JOIN frigate_events f ON f.brivo_access_log_id = b.id
WHERE b.event_type = 'access_granted'
  AND b.access_point = 'park_door'
  AND b.occurred_at > NOW() - INTERVAL '7 days'
  AND f.id IS NULL
ORDER BY b.occurred_at DESC;

  • customers.brivo_sync_error — last error on a per-customer sync attempt
